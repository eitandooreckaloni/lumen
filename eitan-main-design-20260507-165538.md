## Problem Statement

Junior developers entering the Israeli job market for the first time are stuck in a
CV-first lottery. The real problem isn't finding jobs — it's that the user doesn't
know what they want, and every tool optimizes the wrong thing (volume, not clarity).
The deeper problem: the networking-first approach is objectively better, but users
fall back to CV submission because it's less scary and requires less accountability.
Lumen's job is to keep them on the networking track — not just help them once, but
hold them through the whole process.

## What Makes This Cool

- The "guide who holds you accountable" framing: not a search tool, not an assistant,
  but a coach that pings you the morning after a meeting and asks "how did it go?"
- The Mom Test for job search: behavioral interview before any market suggestion
- The Israeli cultural fit: Israel runs on warm intros and network, which makes the
  conversational approach native rather than novel
- The editorial character of the agent: honest, bounded, not a chatbot
- Personal origin: built to help a few friends navigate a scary process

## Constraints

- Cost-conscious: no paid LinkedIn API; Claude API as the primary cost driver
- No autonomous send of LinkedIn messages — user always clicks Send on LinkedIn; agent drafts only
- Israel-only v1: not a geographic wedge, Israel IS the product
- No LinkedIn API, no extension, no scraping — ever. People-surfacing uses
  URL construction only: Lumen builds a LinkedIn people search URL from the person's
  name + company, user clicks, results appear in a new tab (user is already logged in).
- Send flow: draft in Lumen → copy-button modal (see Stage 4/5 spec) → user opens
  recipient's LinkedIn profile and sends. Compose pre-fill with member ID is aspirational
  (spike in Sprint 0); copy-button modal is the default.
- Friendly and fun tone; the product is personal, not enterprise

## Premises

1. The core job is "keep me off the CV treadmill and on the networking track."
   Value is recurring, not single-session. The product has to find the user,
   not wait for them to open the app.

2. 5 behavioral Mom Test questions may not be enough for juniors with thin
   experience. Need a fallback path when the user genuinely has limited signal
   (e.g., just started coding, no projects, straight out of school).

3. People-surfacing needs no extension, no API, no scraping — ever. LinkedIn's
   public people search URL pattern (`?keywords=name+company`) is enough. Lumen
   constructs the search URL; user clicks and picks the right person. The send
   flow (copy-button modal) is a separate, independent problem from finding people.

4. The scariest step is the first message. The product earns trust by making
   that step feel less scary — not by automating it away, but by making the draft
   so good that the user wants to send it.

5. Israel is the whole focus. The Israeli startup ecosystem, the warm-intro culture,
   the army/uni connections — these are features, not constraints.

## Stages (v1 scope)

- **Stage 1: Mom Test Interview** — Constraint capture + 5 behavioral questions
  + hypothesis synthesis. Fallback path for thin profiles (vicarious experience mode).
- **Stage 2: Market Mapping** — Claude-powered Israeli company suggestions, filtered
  by constraints. User swipes to keep/skip. Seeded with static JSON for reliability.
- **Stage 3: People Search** — No extension, no API. Lumen constructs a LinkedIn
  people search URL from name + company (e.g., `linkedin.com/search/results/people/
  ?keywords=Ori+Goshen+AI21+Labs`). User clicks the chip, LinkedIn search opens in
  a new tab — they're already logged in, results appear immediately.
  **LinkedIn result hint:** Displayed beneath the chip before the user clicks:
  Claude generates a one-line tip from the user's hypothesis + company data, e.g.
  "Look for someone with 2+ years at {company} who posts about {domain from hypothesis}."
  Helps junior devs identify the right person from noisy search results.
  User picks the right person, pastes their profile URL back into Lumen. Lumen stores
  the contact and generates a hook (1–2 sentences: why this person is worth reaching
  out to given the user's hypothesis, and the angle to use in the message).
- **Stage 4: Draft Generation** — Lumen drafts the outreach message in the user's
  voice using **Niche + Humble + CTA** shape (concrete definition):
  - **Niche:** One specific reason this person is relevant to the user's hypothesis —
    drawn from Lumen's knowledge of the company, the contact's role, and the user's
    stated interests. Not generic ("you work in tech") but specific ("you built the
    infra layer at AI21 during their model serving rewrite").
  - **Humble:** One sentence acknowledging early-career status and genuine curiosity,
    not asking for a job. "I'm a junior dev who's been obsessing over real-time
    inference and I'd love to hear how you think about the tradeoffs."
  - **CTA:** One low-ask request — 15-minute call, one question about their path, one
    piece of advice. Never "Can we hop on a call?" (too vague). Specific: "15 minutes
    to ask you about X."
  - **Total draft length:** 3 sentences maximum. If the user says the draft doesn't
    sound like them, the agent prompts for one clarifying example and regenerates once.
    No multi-turn correction loop in v1.
- **Stage 5: Send + Reminder Loop** — One-click LinkedIn compose pre-fill (or copy
  modal fallback). "I sent it" logs the send and schedules a follow-up reminder email
  (5–7 days). Post-meeting reflection: agent asks 2 questions ("what interested you,
  what didn't?") and updates the hypothesis + open questions accordingly. Reflection
  flow is v1-minimal — no structured UI, just a short chat exchange.
  **No-reply flow:** When the reminder fires and the user logs "no reply after 1 week,"
  Lumen shows: "No reply is normal — most people don't respond to first messages.
  Try one more? Or move on to a new contact?" Two options: "Draft a follow-up"
  (generates a shorter, softer second message as `Message.type = follow_up`) or
  "Move on" (marks Contact.status = no_reply and surfaces the next Keep'd company
  from Stage 2). Completes the loop at the most common outcome.

## Approaches Considered

### Approach A: Coach First (WhatsApp/email bot)
Minimum viable accountability — a bot that runs Mom Test over WhatsApp and fires
post-meeting follow-ups. No UI. Ships in a weekend.
Completeness: 3/10 — validates retention only, not the full thesis.

### Approach B: Full Flow, Manual People (chosen)
Complete 5-stage conversation as a web app. Claude handles interview + drafting.
Users find and paste names/LinkedIn URLs; Lumen drafts the message. One-click
compose pre-fill for send. No extension needed.
Completeness: 7/10 — delivers the complete value loop as a v1 scope.

### Approach C: Warm Intro Engine
Deferred to v2. Start from user's LinkedIn connections CSV export; map warm paths
to target companies; draft intro requests to mutual connections.

## Recommended Approach

**Approach B: Full Flow, Manual People.**

Build the complete 5-stage loop. Key design decisions:

### 1. Proactive outreach is the retention mechanism
The product reaches out, not waits. Every "I sent it" action schedules a
follow-up reminder email that fires N days later whether or not the user opens
the app. Trigger: cron job fires at 9am on day N (default 5 days after send).
Email content: one sentence — "How did your message to {name} go? Click to log
the reply." — links back to the contact in Lumen.

For v1: email via Resend free tier (3,000 emails/month free, sufficient for early users).
v2 direction: WhatsApp or calendar-aware check-ins (not in scope for v1).

### 2. People search via URL construction — no extension needed
LinkedIn people search has a public URL pattern: `linkedin.com/search/results/people/
?keywords={name}+{company}`. Lumen constructs these URLs server-side from the company
and role data it already has. User clicks a chip, sees real LinkedIn results in a new
tab, picks the right person, pastes their profile URL back. No extension, no API,
no scraping, nothing to install. This is the permanent people-finding solution, not
a v1 workaround.

### 3. Send flow — TECHNICAL SPIKE REQUIRED FIRST (Stage 5 only)
The compose pre-fill is a separate concern from finding people. Send button intends
to open `linkedin.com/messaging/compose?recipient={memberID}&body={draft}`.
**This requires a LinkedIn numeric member ID, not the profile slug.** It is unknown
whether the member ID can be reliably derived from a public profile URL without a
LinkedIn session.

**Sprint 1 action:** Spike this before building the send UI. If the compose URL works,
implement it. If not, the copy-button modal is the primary path.

**Copy-button modal spec** (the probable default): full-screen overlay showing the
draft message in a scrollable textarea (editable), a large "Copy message" button
(copies to clipboard), and a "Open LinkedIn profile →" link that opens the
recipient's LinkedIn URL in a new tab. Below: "Once you've sent it, come back and
click 'I sent it' here." The modal has no close-without-action penalty — the user
can close it and come back.

### 3. Mom Test fallback for thin profiles
Baseline: 5 behavioral questions (last project that kept you up past midnight, what
specifically pulled you in, what bored you, a class that didn't land, what you've
learned in free time).

Fallback trigger: if ≥ 2 answers are "I don't know" / "I haven't done much" / "nothing
really" — agent switches to vicarious mode:
- "Tell me about someone whose work you follow and wish you were doing."
- "What's the last thing you read about technology that you actually wanted to tell someone?"
- "If you had a month with no obligations, what would you build or learn?"

Still behavioral, just uses observed interest instead of direct experience.

### 4. Anti-ChatGPT boundary
Implementation: system prompt instruction + UI affordance.
- System prompt: "You are Lumen, a networking advisor for Israeli juniors. You are
  scoped to the job search and networking process. If the user asks about anything
  outside this scope (career advice in general, coding help, general questions),
  respond with exactly: 'I'm scoped to the networking work. Happy to come back to
  that whenever.' Do not elaborate."
- UI: chat input placeholder text reads "What's on your mind about the search?"
  The input is not a general chat box — it appears inside the context of the active
  stage. Stage-specific prompts guide the conversation forward.

### 5. Market mapping: Claude + static seed
"Claude knows Israeli companies well enough" is an untested claim for companies
founded after 2023. Mitigation for v1:
- Seed a static JSON of 100–150 Israeli tech companies (name, stage, HQ city,
  sub-theme tags, LinkedIn URL). Sources: Startup Nation Central public data,
  GeekTime, IVC Research Center public lists.
- Claude uses the seed as a retrieval source + adds its own knowledge for companies
  not in the seed.
- Display a disclaimer on suggested companies: "Verify this information — company
  details may have changed."
- Market mapping swipe UI: show 5 companies at a time as cards, two buttons (Keep /
  Skip), no undo. After all 5 are swiped, show 5 more (up to ~30 total per run).
  Empty state: "Only 3 strong matches for your constraints — try relaxing one
  constraint or exploring an adjacent theme."
- Example hypothesis output: "You seem drawn to the infrastructure and performance
  layer of AI systems, not consumer-facing products. Best fit with platform and
  backend-infra teams at Series A–B Israeli AI startups."

## Minimal Data Model

```
User         id, email, created_at
Interview    id, user_id, constraints (JSON), questions_answers (JSON), hypothesis (text), created_at
Contact      id, user_id, name, linkedin_url, company, role, hook (text), status (enum: drafted/sent/replied/no_reply)
Message      id, contact_id, draft_text, sent_at, type (enum: initial/follow_up)
Reminder     id, contact_id, scheduled_at, type (enum: follow_up), fired_at
```
Note: `Note` table deferred to v2 (paste/voice_memo notes feature not in v1 scope).
`Contact.draft` removed — working draft is always a `Message` record.
`Contact.status` reflects the latest `Message` outcome, not a direct field — query
for the most recent Message record to determine effective status.
v1 supports multiple messages per contact: `Message.type = initial` for the first
outreach, `Message.type = follow_up` for subsequent messages.

## Technical Architecture

- **Framework:** Next.js (App Router) + Vercel
- **Auth:** Auth.js v5 + Resend email provider. Magic-link email flow: Resend sends
  the link, Auth.js handles token generation, one-time-use enforcement, session cookie
  (httpOnly, Secure, SameSite=lax, 7-day expiry), and logout. No custom crypto needed.
- **Rate limiting:** Upstash Rate Limit middleware (free tier: 10,000 req/day). Applied
  to the magic-link request endpoint and all Claude streaming routes. Also provides
  Redis for per-session cost budgets.
- **Database:** Vercel Postgres + Drizzle ORM with drizzle-kit migrations. (See TODOS-1.)
- **AI:** Anthropic Claude API (claude-sonnet-4-6), streaming via Next.js API route
  using `ReadableStream`. `max_duration: 60` on the Vercel function config.
  - **Prompt caching:** Enabled on system prompts (identical per stage = high cache hit
    rate). Reduces input token cost by ~90% on cached portions.
  - **Per-user daily budget:** Upstash Redis counter. Limit: 50 Claude calls/user/day.
    At cap: show "You've been busy! Come back tomorrow." (prevents runaway cost.)
  - **Error handling:** Client handles three explicit failure states — (1) 60s timeout:
    "Taking longer than expected — tap to retry", (2) Claude API error (429/500):
    "Something went wrong, try again", (3) mid-stream abort: preserve partial text
    + retry affordance.
- **Conversation history:** On stage transition (e.g., Mom Test → Market Mapping),
  Claude generates a 3–5 sentence summary of the completed stage. This summary is
  prepended to the system prompt for all subsequent stages. Only the current stage's
  turn history is passed in the `messages` array, **capped at 20 turns per stage**
  (at cap: Claude generates early summarization and forces stage transition).
- **Vercel tier:** Hobby plan is fine for 2–3 friends. Upgrade to Pro (~$20/mo) when:
  more than 5 concurrent users expected, streaming timeouts appear in logs, or ready
  to share beyond the initial friend group. (See TODOS-3.)
- **Email:** Resend free tier for transactional email (magic-link + reminders)
- **Cron:** Vercel Cron (Hobby: up to 100 cron jobs per project, max 1 execution/day
  each — updated Jan 2026). Daily sweep fires at 9am. Handler queries
  `SELECT * FROM reminders WHERE scheduled_at <= now() AND fired_at IS NULL`,
  sends one reminder email per result, then sets `fired_at = now()` in the **same
  transaction** (prevents duplicate sends on cron re-runs). If Resend returns 429,
  do NOT set `fired_at` — retry on next cron run.
- **Deployment:** GitHub → Vercel (push to main deploys automatically)

## Open Questions

- Mom Test fallback threshold — ≥2 thin answers is the trigger; validate this
  feels right after first user tests.
- LinkedIn member ID resolution — spike required (see Send UX above).
- Hebrew drafts for v2 — will Claude draft in Hebrew naturally if instructed? Test
  separately before designing the v2 language flow.

## Success Criteria

- A friend completes the onboarding interview and has a hypothesis + a draft message
  ready to send in under 20 minutes
- A friend sends at least 3 messages in their first week using Lumen
- When a user receives a reply, they come back to Lumen to log it and get the
  next step (rather than handling it alone in LinkedIn)
- Zero "I just use it like ChatGPT" complaints from the first cohort
